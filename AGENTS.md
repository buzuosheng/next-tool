# 项目 AI 协作说明（前端武器库）

## 项目概览

- **技术栈**：Next.js 15（App Router）、React 19 RC、TypeScript、Tailwind CSS、shadcn/ui 组件
- **站点语言**：简体中文（`<html lang="zh-CN">`）
- **产品定位**：面向前端 / 开发者的在线小工具集合（JSON 格式化、UUID 生成、时间戳转换、IP 查询、二维码等）

在本项目中，AI 助手的主要职责是：**扩展/改进各类工具页面的功能与体验，同时保持一致的 SEO、UI/UX 与代码质量标准。**

## 通用协作规则

1. **语言与文案**
   - 默认使用简体中文进行文案和注释（技术名词、协议名等可保留英文）。
   - 面向用户的描述应简洁、具体，避免堆砌关键字。
2. **代码风格**
   - 使用函数式组件与 Hooks，遵循现有文件结构与命名风格。
   - 优先复用已有组件（`Card`、`Button`、`Input`、`Table`、`Popover` 等），避免重复造轮子。
3. **结构约定**
   - 站点布局统一由 `app/layout.tsx` 控制，不在子页面重复引入站点级头部/脚部。
   - 新工具优先采用「路由页 + 业务组件」模式：`app/<tool>/page.tsx` 引用 `components/tools/*Tool.tsx`。

## Next.js 与路由约定

- **App Router**：所有页面均使用 App Router 约定式文件结构，避免混用旧的 `pages` 目录。
- **数据获取**
  - 工具本身多为纯前端逻辑，优先使用 Client Component。
  - 涉及外部 API（如 IP 查询）时，推荐通过 Server Component 或 Route Handler 代理，避免在前端泄露敏感信息。
- **日志与调试**
  - 禁止在生产逻辑中残留 `console.log` 等调试输出（如 IP 页面中的日志），必要时使用受控的日志方案。

## SEO 约定

1. **Metadata**
   - 每个工具页面都应导出 `export const metadata: Metadata = { ... }`。
   - 标题推荐格式：`<工具名称>_前端武器库`，例如：`"JSON 格式化_前端武器库"`。
   - `description` 清晰描述该工具能做什么、适合谁使用，不要仅罗列关键字。
   - `keywords` 覆盖：工具名（中/英）、“在线工具”“前端工具”“开发工具”等。
2. **标题结构**
   - 每个页面仅保留一个语义上的 `<h1>`，与该工具主标题一致。
   - 副标题与说明使用 `<h2>~<h4>` 或普通文本，避免跳级和过度使用标题标签做样式。
3. **URL 与站点结构**
   - 工具路径保持简洁、语义化，例如 `/json`、`/uuid`、`/ip`。
   - 如未来增加 `robots.txt` 与 `sitemap`，优先使用 Next.js 官方约定（`app/robots.txt`、`app/sitemap.ts`）。

## UI / UX 与可访问性

1. **视觉与布局**
   - 保持当前风格：居中布局、适度留白、卡片式内容块、柔和渐变背景。
   - 在关键断点（375 / 768 / 1024 / 1440）下避免横向滚动和重要内容溢出。
2. **交互反馈**
   - 所有可点击元素（按钮、卡片、图标按钮）必须提供明显的 hover 和 focus 状态。
   - 使用 shadcn/ui 的 `Button`、`Input` 等组件时，沿用其内置的 `focus-visible` 样式，避免手动移除 `outline` 而不补充替代。
3. **可访问性（结合 Web Interface Guidelines）**
   - 图标按钮（仅图标无文字）必须提供 **可访问名称**：
     - 优先增加可见文字；
     - 否则使用 `aria-label` 或 `sr-only` 文本（例如复制、重置、刷新等按钮）。
   - 所有 `Image` 必须设置合适的 `alt` 文本（装饰性图片使用 `alt=""`）。
   - 表单控件始终搭配 `Label`，并通过 `htmlFor` 或包裹元素正确关联。
   - 避免在键盘交互路径上出现「只能鼠标点击」的操作。

## 表单与输入体验

- 输入框与按钮组合时，保证：
  - 标签清晰描述用途（如“输入时间戳”“输入 IP 地址或域名”）。
  - 占位符给出示例而非重复标签（例如 `YYYY-MM-DD HH:mm:ss`）。
  - 错误信息贴近字段展示，必要时配合 Toast。
- 涉及复制到剪贴板的操作，统一：
  - 通过按钮点击触发；
  - 显示 Toast 提示（已复制 / 出错原因）。

## 文案与多语言

- 站点当前默认面向中文用户：
  - 新增页面与组件应优先使用中文文案；
  - 现有英文文案（如 “Your Text”“Result”“Decimal Converter” 等）在调整时应统一为中文或中英双语。
- 标题与按钮文案遵循：
  - 使用动词开头，表达清晰操作（如“生成二维码”“复制结果”）。
  - 避免模糊标签（如“确定”“继续”），改为具体动作。

## 安全与外部服务

- 请求第三方接口时：
  - 明确错误分支，向用户展示友好提示，而不是原始错误消息。
  - 如果未来引入鉴权或付费 API，优先将请求下沉到 Server 侧。

## AI 修改 / 新增代码时的检查清单

- [ ] 是否为新页面添加或更新了合理的 `metadata`（标题、描述、关键字）。
- [ ] 是否保持了现有的布局与视觉风格（容器宽度、间距、字体大小）。
- [ ] 是否为新增的图标按钮提供了 `aria-label` 或 `sr-only` 文本。
- [ ] 是否遵循 App Router 与组件拆分约定（页面负责路由与布局，业务逻辑放在 `components/tools`）。
- [ ] 是否保持用户可见文案使用简体中文（除技术名词外）。
